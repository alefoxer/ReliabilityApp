# Справочник по файлам и модулям проекта

Этот документ описывает каждый значимый файл проекта: зачем он нужен, какие классы и функции содержит, откуда получает данные и куда передает результат. Его цель — помочь новому разработчику быстро понять структуру приложения и безопасно вносить изменения.

## 1. Точка входа и главное окно

### `gui_main.py`

**Назначение:** точка запуска GUI и главный контейнер приложения.

**Роль в системе:** создает `MainWindow`, подключает графический редактор и калькулятор, задает меню, статус-бар, боковую навигацию и общую тему.

**Ключевые элементы:**

- `MainWindow` — главное окно приложения.
- `create_menu()` / похожие внутренние методы меню — собирают действия `Файл`, `Анализ`, `Вид`, `Справка`.
- обработчики действий меню — вызывают методы активного модуля: расчет, формула, экспорт, справка.
- связь `editor.scheme_calculated` с калькулятором — передает результат расчета схемы в модуль калькулятора.

**Связи:**

- импортирует `ModuleVisualRBD` из `gui_visual_editor.py`;
- импортирует `ModuleUniversalCalc` из `gui_calculator.py`;
- использует `theme_manager.py`, `ui_styles.py`, `screen_utils.py`, `logging_utils.py`;
- показывает пользовательские сообщения через Qt.

**Когда менять:** при изменении навигации, меню, статуса, глобального масштаба интерфейса, заголовка окна или подключения новых модулей.

## 2. Графический редактор

### `gui_visual_editor.py`

**Назначение:** визуальное построение структурной схемы надежности.

**Роль в системе:** это основной UI-модуль схемы. Он отвечает за рабочую доску, блоки, линии, шаблоны, сохранение/загрузку, генерацию формулы, расчет по схеме и встроенную самопроверку.

**Ключевые классы:**

- `StepSpinWidget` — компактный числовой ввод с кнопками уменьшения/увеличения.
- `ConnectionLine` — графический объект связи между двумя блоками. Хранит ссылки на исходный и конечный блок, порты, путь линии и подпись. Перестраивает геометрию при перемещении блоков.
- `RBDBlock` — графический блок схемы. Отвечает за отображение, порты, параметры, контекстное меню, уведомление линий о перемещении.
- `GraphicsView` — рабочий вид сцены: zoom, pan, сетка, режим обновления viewport.
- `ModuleVisualRBD` — главный виджет редактора, панели управления, действия пользователя и преобразование сцены в модель.

**Ключевые функции/методы:**

- `to_scheme_model()` — собирает текущую графическую сцену в `SchemeModel`.
- `load_scheme_model()` — восстанавливает графическую сцену из `SchemeModel`.
- `add_block()` — добавляет новый блок.
- `calc_graph()` — запускает расчет текущей схемы через `scheme_adapter.py`.
- `show_formula_dialog()` — строит и показывает формулу по схеме.
- `refresh_summary()` — обновляет левый информационный блок.
- `run_self_check()` — проверяет генератор формул на встроенных шаблонах.
- `save_to_file()` / `load_from_file()` — работают с JSON через `scheme_storage.py`.
- `export_png()` / `export_svg()` — вызывают `scene_exporters.py`.

**Связи:**

- получает/возвращает `SchemeModel`, `BlockModel`, `ConnectionModel` из `rbd_models.py`;
- валидирует через `validators.py`;
- строит формулу через `graph_formula_builder.py`;
- выбирает метод через `scheme_method_selector.py`;
- считает через `scheme_adapter.py`;
- сохраняет/загружает через `scheme_storage.py`;
- экспортирует изображение через `scene_exporters.py`;
- берет шаблоны из `library_templates.py`.

**Когда менять:** при изменении UI редактора, поведения блоков/линий, логики конвертации сцены в модель, встроенной самопроверки и действий правой/левой панели.

## 3. Калькулятор

### `gui_calculator.py`

**Назначение:** универсальный ручной калькулятор методов F1.1-F7.2.

**Роль в системе:** строит форму параметров по выбранной методике, вызывает расчетную функцию, показывает график и результаты, экспортирует отчет и график.

**Ключевые классы:**

- `PlotAndFormulaWidget` — matplotlib-график и область формул.
- `ParamWidget` — числовой параметр с вводом, слайдером и step-кнопками.
- `ModuleUniversalCalc` — основной виджет калькулятора.

**Ключевые методы:**

- `build_dynamic_form()` — перестраивает форму при смене метода.
- `_compact_param_label()` — показывает в форме краткое обозначение параметра.
- `_format_method_details()` — формирует раскрываемый паспорт метода.
- `_prepare_call_kwargs()` — собирает аргументы для backend-функции.
- `calculate()` — запускает расчет и обновляет результаты.
- `export_report()` — сохраняет отчет через `report_exporters.py`.
- `export_plot()` — сохраняет график.
- `receive_scheme_result()` — принимает результат из графического редактора.

**Связи:**

- методы и формулы: `normative_methods.py`;
- описания параметров: `method_metadata.py`;
- расчет: `dependability_backend.py`;
- отчеты: `report_exporters.py`;
- выбор пути: `file_service.py`;
- модели: `rbd_models.py`.

**Когда менять:** при добавлении новых методов, изменении формы ввода, отображения паспортов методик, графиков, экспорта отчета или шаблонов компонентов.

## 4. Диалоги

### `gui_dialogs.py`

**Назначение:** набор вспомогательных диалогов.

**Роль в системе:** отделяет настройки блоков, параметры отчетов и работу со справочником от основных виджетов.

**Ключевые элементы:**

- диалог параметров блока;
- диалог параметров отчета;
- диалоги выбора/редактирования шаблонов.

**Связи:**

- вызывается из `gui_visual_editor.py` и `gui_calculator.py`;
- работает с данными блоков и отчетов;
- может использовать `db_modules.json` для пользовательских шаблонов.

**Когда менять:** при расширении набора параметров блока, структуры отчета или интерфейса шаблонов.

## 5. Модели данных

### `rbd_models.py`

**Назначение:** единый набор dataclass-моделей.

**Ключевые классы:**

- `BlockModel` — блок схемы.
- `ConnectionModel` — связь между блоками.
- `SchemeModel` — полная схема.
- `FormulaInfo` — формула для UI и отчетов.
- `CalculationResult` — результат расчета.
- `ReportData` — данные отчета.
- `ValidationMessage`, `ValidationResult` — результат валидации.

**Связи:** используется почти всеми подсистемами: редактором, формулами, расчетом, экспортом, тестами.

**Когда менять:** только при необходимости расширить общий контракт данных. После изменения нужно проверить сериализацию, генератор формул, экспорт и тесты.

## 6. Генератор формул

### `graph_formula_builder.py`

**Назначение:** центральный движок генерации общей формулы по структурной схеме.

**Ключевые классы:**

- `FormulaExpr` — AST-узел формулы.
- `NormalizedScheme` — нормализованная расчетная схема.
- `FormulaGenerationResult` — полный результат генерации.

**Ключевые функции:**

- `build_formula_for_scheme(scheme)` — возвращает `FormulaInfo` для UI.
- `build_formula_report(scheme)` — возвращает полный формульный отчет.
- `normalize_scheme(scheme)` — выделяет достижимый подграф.
- `extract_reachable_subgraph(scheme)` — совместимая точка вызова нормализации.
- `detect_series_parallel_structure(scheme, metric)` — строит AST по топологии.
- `build_reliability_formula(scheme)` — AST для `P`.
- `build_availability_formula(scheme)` — AST для `Kg`.
- `render_symbolic_formula(expr)` — человекочитаемая формула.
- `render_computable_formula(expr)` — вычислительная строка.
- `evaluate_formula(expr, values)` — численная подстановка.
- `build_formula_explanation(expr, symbols)` — объяснение построения.

**Внутренняя логика:**

- pass-through блоки: вход, выход, junction;
- расчетные блоки становятся символами;
- последовательность — `series`;
- ветвление — `parallel`;
- резервирование — `reserve`;
- вложенные схемы обрабатываются рекурсивно.

**Связи:**

- получает `SchemeModel` из редактора или тестов;
- возвращает `FormulaInfo` в UI;
- используется `scheme_method_selector.py` как единый источник признаков;
- используется `scheme_adapter.py` для расчета по формуле.

**Когда менять:** при исправлении алгоритма формул, добавлении нового типа соединения, поддержке новых метрик или улучшении рендера.

### `formula_library.py`

**Назначение:** библиотека типовых проектных формул, с которыми интеллектуальный генератор сопоставляет фрагменты схемы.

**Ключевые классы и данные:**

- `FormulaDefinition` — описание формулы: идентификатор, название, источник, показатель, тип фрагмента, применимость, ограничения, параметры, общий и вычислительный вид.
- `FORMULA_LIBRARY` — словарь доступных формул.
- `PROJECT_SOURCE` — явная пометка для формул, которые являются методическими формулами проекта, а не отдельной ГОСТ-ссылкой.

**Ключевые функции:**

- `formula_for_fragment(kind, metric)` — возвращает подходящую формулу для типа фрагмента и показателя.

**Связи:**

- используется `intelligent_formula_generator.py`;
- дополняет результат `graph_formula_builder.py`;
- тестируется в `tests/test_intelligent_formula_generator.py`.

**Когда менять:** при добавлении новой типовой формулы, нового показателя или уточнении применимости/ограничений формул.

### `intelligent_formula_generator.py`

**Назначение:** интеллектуальный слой генератора формул. Модуль анализирует AST, выбирает формулы из `formula_library.py`, подставляет параметры схемы и добавляет объяснение выбора.

**Ключевые классы:**

- `FormulaCandidate` — формула-кандидат для фрагмента.
- `FragmentFormulaSelection` — выбранная формула, подстановки и численный результат для одного фрагмента.
- `IntelligentFormulaGenerationResult` — итог анализа, выбранные формулы, обозначения и предупреждения.

**Ключевые функции:**

- `generate_intelligent_formula(scheme, time_horizon, base_report)` — строит расширенный результат по схеме.
- `enhance_formula_info(scheme, formula, base_report, time_horizon)` — добавляет интеллектуальные разделы к `FormulaInfo`, который уже использует UI.

**Связи:**

- получает AST из `graph_formula_builder.py`;
- берет формулы из `formula_library.py`;
- использует `SchemeModel` и `FormulaInfo` из `rbd_models.py`;
- отображается в `gui_visual_editor.py` через стандартное окно формулы;
- тестируется в `tests/test_intelligent_formula_generator.py`.

**Когда менять:** при изменении логики выбора формул, подстановки параметров, объяснения или численной проверки фрагментов.

## 7. Выбор методики

### `scheme_method_selector.py`

**Назначение:** анализ схемы и рекомендация метода расчета.

**Ключевые классы:**

- `SchemeAnalysisResult` — признаки схемы.
- `ScenarioCandidate` — возможный сценарий.
- `MethodCandidate` — кандидат метода.
- `MethodSelectionResult` — итог выбора.

**Ключевые функции:**

- `select_method_for_scheme(scheme)` — полный pipeline выбора.
- `analyze_scheme(scheme)` — извлечение признаков.
- `build_scenario_candidates(analysis)` — список сценариев.
- `build_method_candidates(analysis)` — список методов.
- `recommend_method(methods)` — выбор лучшего кандидата.
- `format_method_selection_html(result)` — HTML для UI.
- `format_method_selection_text(result)` — текст для буфера/отчета.

**Связи:**

- использует `graph_formula_builder.py`;
- читает реестр `normative_methods.py`;
- отображается в `gui_visual_editor.py`;
- тестируется в `tests/test_scheme_method_selector.py`.

**Когда менять:** при добавлении сценариев, правил применимости методов или изменении логики рекомендаций.

## 8. Расчетное ядро

### `dependability_backend.py`

**Назначение:** функции расчета F1.1-F7.2.

**Ключевые функции:**

- `f11` ... `f15` — невосстанавливаемые системы и резервирование.
- `f21` ... `f27` — восстанавливаемые системы.
- `f31` — периодический контроль.
- `f41` — смешанная структура с контролем.
- `f51` — допустимое время простоя.
- `f61` ... `f63` — структурная надежность.
- `f71`, `f72` — структуры с покрытием.

**Вспомогательные функции:** комбинаторика, защита от переполнения, расчет внутренних сумм и вероятностей.

**Связи:**

- вызывается из `gui_calculator.py`;
- реестр вызовов задается в `normative_methods.py`;
- тестируется в `tests/test_backend.py`.

**Когда менять:** при исправлении или добавлении расчетных формул. После изменения обязательно обновить методические описания и тесты.

### `dependability_graph.py`

**Назначение:** графовые и имитационные расчеты надежности.

**Роль:** поддерживает расчет по графу и Monte Carlo. Используется как дополнительное вычислительное ядро для схемных расчетов.

**Когда менять:** при оптимизации Monte Carlo, добавлении новых алгоритмов графового анализа или переносе тяжелых расчетов в worker.

### `scheme_adapter.py`

**Назначение:** адаптер между `SchemeModel` и расчетом.

**Ключевая функция:**

- `calculate_scheme_reliability()` — валидирует схему, строит формулу, выбирает метод, рассчитывает показатели и возвращает `CalculationResult`.

**Связи:**

- `validators.py`;
- `graph_formula_builder.py`;
- `scheme_method_selector.py`;
- `rbd_models.py`;
- `gui_visual_editor.py`.

**Когда менять:** при изменении того, как схема превращается в численный результат.

## 9. Реестр методов и описания

### `normative_methods.py`

**Назначение:** структурированный реестр F1.1-F7.2.

**Ключевой класс:**

- `NormativeMethodSpec` — код метода, название, функция backend, режим ввода, аргументы, формулы, источник, применимость, ограничения, поля результата.

**Ключевые данные/функции:**

- `SUPPORTED_METHODS`;
- `SUPPORTED_METHODS_BY_NAME`;
- `SUPPORTED_METHODS_BY_CODE`;
- `supported_method_names()`;
- `get_method_spec()`.

**Связи:**

- UI калькулятора берет отсюда методы;
- `scheme_method_selector.py` берет кандидатов;
- `tests/test_normative_methods.py` проверяет полноту.

### `method_metadata.py`

**Назначение:** подробные пользовательские описания методов и параметров.

**Ключевые структуры:**

- `MethodGuide`;
- `ParameterDoc`;
- `METHOD_GUIDES`;
- `PARAMETER_DOCS`.

**Когда менять:** если нужно улучшить текст методики, подсказки параметров или добавить новый аргумент.

### `app_constants.py`

**Назначение:** общие константы, тексты и формульные строки исходного проекта.

**Связи:** может использоваться UI и расчетными модулями для единых подписей и справочных данных.

**Когда менять:** при централизованном изменении текстов, подписей и констант, если они не относятся к конкретной методике.

## 10. Сохранение, экспорт, файлы

### `scheme_storage.py`

**Назначение:** JSON-сохранение и загрузка схем.

**Ключевые функции:**

- `save_scheme(path, scheme)`;
- `load_scheme(path)`.

**Связи:** используется `gui_visual_editor.py`, тестируется `tests/test_scheme_storage.py`.

### `scene_exporters.py`

**Назначение:** экспорт графической сцены в SVG и PNG.

**Ключевые функции:**

- `export_scene_to_png(scene, path)`;
- `export_scene_to_svg(scene, path)`.

**Важная логика:** PNG строится по реальным границам `scene.itemsBoundingRect()`, а не как скриншот видимой области.

### `report_exporters.py`

**Назначение:** экспорт отчетов.

**Ключевые функции:**

- `report_to_html(report)`;
- `export_html(path, report)`;
- `export_txt(path, report)`;
- `export_pdf(path, report)`;
- `export_docx(path, report)`;
- `export_xlsx(path, report)`.

**Связи:** вызывается из калькулятора, принимает `ReportData`.

### `file_service.py`

**Назначение:** централизованный выбор пути и уведомления о сохранении.

**Ключевые элементы:**

- `SaveFormat`;
- `choose_save_path()`;
- `notify_save_result()`.

**Зачем нужен:** чтобы логика файловых диалогов не была размазана по UI-коду.

### `db_modules.json`

**Назначение:** пользовательские или демонстрационные шаблоны компонентов/модулей.

**Когда менять:** через UI шаблонов или вручную при подготовке набора примеров.

### `ReliabilityApp.spec`

**Назначение:** спецификация PyInstaller для будущей сборки exe.

**Связи:** подключает иконку `resources/app_icon.ico` и параметры сборки.

## 11. Шаблоны и ресурсы

### `library_templates.py`

**Назначение:** встроенные демонстрационные схемы.

**Ключевые функции:**

- `built_in_templates()`;
- `save_user_templates()`;
- `load_user_templates()`.

**Связи:** используется графическим редактором и самопроверкой.

### `resources/`

**Назначение:** статические ресурсы приложения.

**Файлы:** иконка приложения `app_icon.ico` и резервные изображения, если они присутствуют.

## 12. Валидация, логирование, адаптация

### `validators.py`

**Назначение:** централизованная проверка пользовательских данных и схем.

**Ключевые функции:**

- `validate_scheme()`;
- `validate_scheme_file()`;
- проверки числовых диапазонов и структуры.

**Связи:** используется редактором, адаптером расчета и тестами.

### `logging_utils.py`

**Назначение:** настройка системного логирования.

**Роль:** сохраняет технические детали ошибок в `logs/`, чтобы пользователь не видел raw traceback.

### `screen_utils.py`

**Назначение:** адаптация окон под экран.

**Роль:** помогает не открывать окна больше доступного размера.

### `theme_manager.py`

**Назначение:** хранение и применение масштаба/темы интерфейса.

### `ui_styles.py`

**Назначение:** единый stylesheet приложения.

**Когда менять:** при изменении визуального стиля, compact-кнопок, group box, details-блоков и общего масштаба.

## 13. Тесты

### `tests/test_backend.py`

## 15. Как читать проект новому разработчику

Рекомендуемый порядок:

1. `README.md` — понять назначение.
2. `docs/ARCHITECTURE.md` — понять поток данных.
3. `docs/MODULE_REFERENCE.md` — найти нужный файл.
4. `rbd_models.py` — изучить модель данных.
5. `graph_formula_builder.py` — понять центральную формульную логику.
6. `scheme_method_selector.py` — понять выбор метода.
7. `gui_visual_editor.py` или `gui_calculator.py` — изучать UI только после понимания core-логики.

## 16. Что можно менять безопасно

- Тексты документации.
- Описания параметров в `method_metadata.py`.
- Визуальные стили в `ui_styles.py`.
- Шаблоны в `library_templates.py`, если сохраняется корректная `SchemeModel`.

## 17. Что менять осторожно

- `rbd_models.py` — влияет на сериализацию, UI, отчеты и тесты.
- `dependability_backend.py` — влияет на нормативные расчеты.
- `graph_formula_builder.py` — влияет на формулы, метод selector, расчет и отчеты.
- `scheme_method_selector.py` — влияет на рекомендации и объяснения.
- `gui_visual_editor.py` — большой UI-модуль, легко нарушить связь сцены и модели.

## 18. Главные инварианты проекта

- Схема в UI должна преобразовываться в `SchemeModel`.
- Формула, обозначения и объяснение должны строиться из одного AST.
- Блок не должен попадать только в обозначения или только в объяснение.
- Пользователь сам выбирает путь экспорта.
- Расчетные функции не должны зависеть от UI.
- Все пользовательские сообщения должны быть на русском языке.
