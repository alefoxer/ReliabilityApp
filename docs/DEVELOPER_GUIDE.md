# Руководство разработчика

## 1. Запуск в режиме разработки

Установите зависимости:

```bash
pip install -r requirements.txt
```

Запустите приложение:

```bash
python gui_main.py
```

Запустите тесты:

```bash
pytest
```

Если интерпретатор Python не доступен по команде `python`, используйте установленный launcher или полный путь к `python.exe`.

## 2. Архитектурный принцип проекта

Проект сохраняет исходную плоскую структуру файлов, но вокруг нее добавлен мягкий сервисный слой:

- UI-модули отвечают за окна, панели, диалоги и действия пользователя.
- Модели данных описаны в `rbd_models.py`.
- Расчетное ядро остается в `dependability_backend.py`.
- Генератор формул отделен в `graph_formula_builder.py`.
- Библиотека типовых формул отделена в `formula_library.py`.
- Интеллектуальное сопоставление фрагментов схемы с формулами отделено в `intelligent_formula_generator.py`.
- Анализ схемы и выбор методики отделены в `scheme_method_selector.py`.
- Сериализация, экспорт, валидация и темы вынесены в отдельные модули.

Цель такого подхода — не переписывать приложение заново, а сделать существующий код сопровождаемым.

Подробная карта каждого файла, его классов, функций и связей находится в [MODULE_REFERENCE.md](MODULE_REFERENCE.md). Если нужно быстро понять, за что отвечает конкретный файл, начинать лучше с этого справочника.

## 3. Основные модули

### `gui_main.py`

Создает главное окно, меню, статус-бар и переключение между модулями. Здесь подключаются:

- `ModuleVisualRBD`;
- `ModuleUniversalCalc`;
- единый стиль;
- иконка приложения;
- действия меню `Анализ`, `Файл`, `Вид`, `Справка`.

### `gui_visual_editor.py`

Содержит графический редактор. Ключевые классы:

- `GraphicsView` — отображение сцены, zoom, pan, сетка.
- `RBDBlock` — графический блок.
- `ConnectionLine` — объектная линия связи между блоками.
- `ModuleVisualRBD` — основной виджет редактора, панели, действия, расчет и формула.

Редактор преобразует сцену в `SchemeModel` через `to_scheme_model()`. Обратное восстановление выполняется при загрузке JSON или применении шаблона.

### `gui_calculator.py`

Содержит универсальный калькулятор методов F1.1-F7.2:

- выбор метода из `normative_methods.py`;
- динамическая форма параметров;
- компактные обозначения параметров;
- раскрываемый паспорт методики;
- вызов расчетной функции;
- график через matplotlib;
- экспорт отчета и графика.

### `dependability_backend.py`

Расчетное ядро. Здесь находятся функции:

- `f11` ... `f15`;
- `f21` ... `f27`;
- `f31`;
- `f41`;
- `f51`;
- `f61` ... `f63`;
- `f71`, `f72`.

Имена аргументов этих функций нельзя менять без обновления `normative_methods.py`, UI и тестов.

### `normative_methods.py` и `method_metadata.py`

`normative_methods.py` связывает код метода, функцию backend, режим ввода, список аргументов, формулы и поля результата.

`method_metadata.py` содержит человекочитаемые описания:

- `MethodGuide`;
- `ParameterDoc`;
- назначение метода;
- область применения;
- пример;
- расшифровку параметров.

### `graph_formula_builder.py`

Генератор общей формулы по схеме. Он построен вокруг AST `FormulaExpr` и результата `FormulaGenerationResult`.

Основной конвейер:

```text
SchemeModel
→ normalize_scheme()
→ detect_series_parallel_structure()
→ FormulaExpr
→ render_symbolic_formula()
→ render_computable_formula()
→ build_formula_explanation()
→ FormulaInfo
```

### `scheme_method_selector.py`

Интеллектуальный анализ схемы и выбор метода. Модуль строит:

- `SchemeAnalysisResult`;
- `ScenarioCandidate`;
- `MethodCandidate`;
- `MethodSelectionResult`.

Он не выполняет расчет сам, а принимает формульный отчет и признаки схемы, затем выбирает наиболее подходящий подход.

### `scheme_adapter.py`

Связывает редактор и расчет:

1. валидирует схему;
2. строит формулу;
3. выбирает метод;
4. вычисляет показатели;
5. возвращает `CalculationResult`.

### `report_exporters.py`

Единый слой экспорта. Принимает `ReportData` и формирует:

- HTML;
- TXT;
- PDF;
- DOCX;
- XLSX.

DOCX формируется через `python-docx`, XLSX через `openpyxl`, изображения графиков и схем вставляются при наличии корректных PNG-файлов. Эти зависимости указаны в `requirements.txt` и должны быть установлены перед экспортом.

## 4. Модель данных

Главные dataclass-модели находятся в `rbd_models.py`.

`BlockModel`:

- `block_id` — стабильный идентификатор;
- `name` — имя, отображаемое в формуле;
- `kind` — тип блока: вход, выход, функциональный блок, junction;
- `x`, `y` — координаты;
- `params` — параметры надежности;
- `is_subscheme` — признак вложенной схемы;
- `nested_scheme` — внутренняя `SchemeModel`.

`ConnectionModel`:

- `connection_id`;
- `source_id`, `source_port`;
- `target_id`, `target_port`.

`SchemeModel`:

- `name`;
- `blocks`;
- `connections`;
- `metadata`.

`FormulaInfo`:

- `text` — пользовательская формула;
- `latex`;
- `structural`;
- `computational`;
- `steps`;
- `symbols`;
- `used_blocks`;
- `unused_blocks`;
- `warnings`.

`CalculationResult`:

- `method_name`;
- `indicators`;
- `source`;
- `formula`;
- `graph_points`;
- `details`.

`ReportData`:

- контейнер для всех экспортеров отчетов.

## 5. Как добавить новый расчетный метод

1. Добавить функцию в `dependability_backend.py`.
2. Не использовать UI внутри расчетной функции.
3. Вернуть словарь с показателями, например `{"P": ..., "T0": ...}`.
4. Добавить запись `NormativeMethodSpec` в `normative_methods.py`.
5. Добавить описание метода в `method_metadata.py`.
6. Для каждого аргумента добавить `ParameterDoc`.
7. Добавить тест в `tests/test_backend.py` или `tests/test_normative_methods.py`.
8. Проверить калькулятор вручную.

Важно: если формула не подтверждена материалами проекта, метод не должен выдавать фиктивный расчет как нормативный.

## 6. Как расширить генератор формул

Генератор основан на `FormulaExpr`. Чтобы добавить новый тип композиции:

1. Добавить новый `kind` в `FormulaExpr`.
2. Реализовать:
   - `collect_symbols`;
   - `to_sympy`;
   - `evaluate`;
   - `render_pretty`;
   - `render_computable`;
   - `render_structure`.
3. Обновить `_build_segment()` или функцию распознавания структуры.
4. Добавить объяснение в `_explain_expr()`.
5. Добавить тесты в `tests/test_formula_builder.py`.

Инвариант: все пользовательские разделы формулы должны строиться из одного AST, а не из разных списков UI.

### `formula_library.py`

Библиотека формул содержит `FormulaDefinition` и словарь `FORMULA_LIBRARY`. Здесь описаны типовые правила для пустой схемы, простого элемента, последовательного соединения, параллельного соединения, MVP-резервирования и коэффициента готовности. При добавлении новой типовой формулы нужно указать источник, применимость, ограничения, параметры, общий вид и вычислительный шаблон.

### `intelligent_formula_generator.py`

Модуль проходит по AST из `graph_formula_builder.py`, определяет тип каждого фрагмента и выбирает подходящую формулу из `formula_library.py`. Результат добавляется в `FormulaInfo.steps`, `FormulaInfo.structural`, `FormulaInfo.computational` и используется UI/отчетами. Это слой объяснения и подстановки, а не замена базового AST-генератора.

## 7. Как расширить выбор метода

Правила выбора находятся в `scheme_method_selector.py`.

Для нового сценария:

1. Добавить признак в `SchemeAnalysisResult`, если существующих признаков недостаточно.
2. Обновить `_features()` и `_structure_type()`.
3. Добавить `ScenarioCandidate` в `build_scenario_candidates()`.
4. Добавить логику кандидата в `build_method_candidates()` или `_candidate_for_spec()`.
5. Уточнить `recommend_method()`.
6. Добавить тест в `tests/test_scheme_method_selector.py`.

## 8. Как изменить интерфейс

- Главное окно: `gui_main.py`.
- Редактор схем: `gui_visual_editor.py`.
- Калькулятор: `gui_calculator.py`.
- Диалоги: `gui_dialogs.py`.
- Общие стили: `ui_styles.py`.
- Масштаб и адаптация под экран: `screen_utils.py`, `theme_manager.py`.

При изменении UI важно не смешивать расчетную логику с кодом виджетов. Если действие требует вычислений, лучше добавить функцию в сервисный модуль и вызвать ее из UI.

## 9. Сохранение схемы

Сериализация находится в `scheme_storage.py`. Формат основан на dataclass-моделях:

```json
{
  "name": "Новая схема",
  "blocks": [],
  "connections": [],
  "metadata": {}
}
```

При добавлении новых полей нужно обеспечить обратную совместимость: старые JSON-файлы не должны ломать загрузку.

## 10. Экспорт

Путь сохранения выбирается через `file_service.py`. Экспортеры не должны сами спрашивать путь у пользователя: UI выбирает путь, а `report_exporters.py` только записывает файл.

Для нового формата:

1. Добавить функцию `export_<format>(path, report)`.
2. Обновить UI выбора формата.
3. Добавить smoke test.

## 11. Валидация и ошибки

Валидация схемы находится в `validators.py`. Пользовательские ошибки должны возвращаться как `ValidationResult`, а не приводить к traceback в интерфейсе.

Технические детали ошибок логируются через `logging_utils.py` в каталог `logs/`.

## 12. Кодировка

Все новые файлы документации и исходники следует сохранять в UTF-8. Если в терминале видны искаженные русские строки, это проблема кодовой страницы консоли, а не обязательно содержимого файла.

## 13. Рекомендации по сопровождению

- Не удалять существующие публичные классы и функции без необходимости.
- Не менять имена аргументов backend-функций без синхронного обновления реестра методов.
- Не строить формулы строковой конкатенацией в UI.
- Не брать список блоков для обозначений напрямую из canvas; использовать `FormulaInfo.symbols`.
- Новую функциональность закрывать тестами.
- Любое предупреждение пользователя писать на русском языке.
